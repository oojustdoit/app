# 📋 项目设计与实施问题总结及解决方案

在本次 DevOps 项目从零搭建到完整落地的过程中，我们遇到了一系列典型的设计与实施问题，涵盖了容器化、CI/CD、镜像仓库、网络、监控等各个环节。以下是问题汇总及其应对策略，供后续项目参考。

---

## 🔍 问题清单与解决方案

| 阶段 | 问题描述 | 解决方案 |
| :--- | :--- | :--- |
| **容器化** | `Dockerfile` 中 `COPY requirements.txt` 失败，因为文件不存在 | 在项目根目录创建 `requirements.txt` 并写入依赖（如 Flask） |
| **版本控制** | 执行 `git remote` 等命令时报错 `fatal: not a git repository` | 先执行 `git init` 初始化本地仓库，再关联远程仓库 |
| **远程认证** | 使用 HTTPS 推送需要反复输入密码/令牌 | 改用 SSH 密钥认证，生成密钥对并添加到 GitHub 账户，设置 `git remote` 为 SSH 格式 |
| **镜像拉取超时** | 服务器（阿里云）拉取 Docker Hub 镜像超时（`Client.Timeout exceeded`） | 配置 Docker 镜像加速器（如 `docker.xuanyuan.me`），并重启 Docker 服务 |
| **私有仓库拉取** | 未登录 Docker Hub 时拉取私有仓库返回 `not found`（误导性错误） | 执行 `docker login`，使用 Access Token（非密码）登录；或直接将仓库设为公开 |
| **ACR 登录失败** | 尝试使用阿里云 ACR 替代 Docker Hub，但 `unauthorized: authentication required` 反复出现（固定密码未生效/用户名错误） | 最终放弃 ACR，回归 Docker Hub 公开仓库，避免复杂凭证管理 |
| **GitHub Actions 变量传递** | 部署脚本中 `$IMAGE_TAG` 为空，导致镜像标签无效（`invalid reference format`） | 在 `appleboy/ssh-action` 步骤中通过 `env` 显式定义 `IMAGE_TAG: ${{ github.sha }}`，并通过 `envs` 传递到远程脚本 |
| **部署中断** | 蓝绿部署时健康检查不完善，可能造成服务不可用 | 增加前后两次健康检查（新容器启动后和最终切换后），失败则退出脚本，保持旧容器继续服务 |
| **Docker Compose 挂载错误** | 监控部署时 `prometheus.yml` 被创建为目录而非文件，导致挂载失败（`not a directory`） | 删除目录，重新创建为普通文件，确保挂载类型正确 |
| **Grafana 仪表盘导入失败** | 输入的仪表盘 ID（如 `14665`）无效或不存在 | 更换为有效 ID（如 `9688`）或通过官方 JSON 文件导入 |
| **端口访问不通** | 容器运行正常但外部无法访问（`curl localhost` 成功但公网失败） | 检查云平台安全组，开放应用端口（5000）入方向规则 |

---

## 🧩 核心设计缺陷与改进措施

- **镜像仓库选择**：初期尝试阿里云 ACR 导致认证复杂，最终采用 Docker Hub 公开仓库，简化流程，但生产环境建议使用企业级私有仓库（如 Harbor）配合内网拉取。
- **部署策略**：虽然实现了蓝绿部署，但未考虑数据库迁移等状态依赖，大型应用应引入更完善的滚动更新或金丝雀发布。
- **监控集成**：监控组件独立部署，但未与 CI/CD 联动（如部署后自动触发 Grafana 告警测试），后续可增加部署后验证步骤。
- **环境依赖**：未使用 `.env` 文件管理环境变量，多处硬编码 IP 和用户名，后续应统一注入环境变量，提高可移植性。

## ✅ 最终成果

通过逐步排查和优化，项目成功实现了：
- **自动化构建与部署**：每次 `git push` 自动完成镜像构建、推送和服务器更新。
- **高可用部署策略**：蓝绿切换 + 健康检查，最大程度降低服务中断。
- **完善的可观测性**：实时监控应用请求量、延迟、错误率，并提供可视化仪表盘。
- **版本可回溯**：每个 commit 对应唯一镜像标签，方便快速回滚。

这些经验不仅解决了当前项目的问题，也为未来更复杂的 DevOps 实践奠定了坚实基础。
